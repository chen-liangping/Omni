# 📄 需求文档（PRD）

## 1. 背景与目标

* 当前问题：多个分支并行开发，统一部署到 `stg` 环境，容易出现覆盖和冲突，缺乏对“当前环境状态”的可见性。
* 目标：

  1. 清晰展示 **每个环境当前生效的分支**；
  2. 允许将环境与分支进行关联，并附加功能说明；
  3. 在分支合并时自动更新环境状态；
  4. 支持规划分支的生效周期，减少冲突。

---

## 2. 功能模块

### 2.1 环境状态展示

* **需求描述**：在页面上展示所有环境（stg、uat、prod…）当前运行的分支信息。
* **字段**：

  * 环境名称（stg/uat/prod）
  * 当前分支名
  * Commit ID（可跳转到 GitHub commit 页面）
  * 功能说明（由用户配置）
  * 部署时间
  * 部署人
* **交互**：

  * 点击分支名 → 打开分支详情页（包含历史部署记录）。

---

### 2.2 环境与分支关联

* **需求描述**：在环境状态页面，增加“关联”按钮，允许用户手动选择某个分支，并将其部署/绑定到某个环境。
* **配置项**：

  * 环境选择（下拉框：stg/uat/prod）
  * 分支选择（从 GitHub 拉取分支列表）
  * 功能说明（必填，文本）
* **交互**：

  * 用户点击「关联分支」按钮 → 弹出配置弹窗 → 填写后确认 → 更新环境状态表。

---

### 2.3 分支合并触发更新

* **需求描述**：当分支 PR 合并到目标分支（如 `stg`），系统应自动更新环境状态。
* **实现逻辑**：

  * 监听 GitHub Webhook（`push`、`pull_request` 合并事件）。
  * 获取最新 commit ID。
  * 更新对应环境的“当前分支”和“最新 commit”。
  * 自动触发部署流水线（CI/CD）。
* **异常处理**：合并失败时，提示“更新未完成”，并允许手动重试。

---

### 2.4 分支生效时间规划

* **需求描述**：用户可以为某个分支设置预计生效时间、下线时间。
* **字段**：

  * 分支名
  * 生效时间（起始时间）
  * 下线时间（结束时间，可选）
  * 环境（stg/uat/prod）
* **交互**：

  * 支持在日历/Gantt 图上拖拽调整时间。
  * 超过下线时间未合并，系统发出提醒。

# 新增逻辑

详情页变成两个tab，一个tab展示当前内容，生效中和待生效卡片，一个tab展示测试完成的分支名称、功能说明

- 生效中卡片：生效中卡片新增”测试完成“”合并代码“操作，
  - 测试完成：点击后标记为complete。进入测试完成 tab页，在测试完成tab看到分支名称、功能说明，可点击”合并代码“”测试回退“按钮
    - 合并代码：点击后出现弹窗，提示”合并代码将把该分支的代码合并到develop分支，请确认代码已完成功能测试“，点击确认后，生成commit 链接，链接支持复制
    - 测试回退：点击后出现弹窗，提示“测试回退后会重新进入测试流程，可重新规划部署时间，进行功能测试”，点击确认后回退
  - 合并代码：点击后出现弹窗，提示”合并代码将把该分支的代码合并到develop分支，请确认代码已完成功能测试“，系统默认带入分支的功能说明，用户可修改内容。点击确认后，生成 commit 链接，链接支持复制
- 即将生效卡片：新增“立即生效”操作，点击后出现弹窗，提示”确定立即生效吗，确定后环境将立即切换部署分支，请确保原分支已测试完成“，点击确认后立即生效，生效时间改为点击的时间和设置的失效时间

已经合并代码或测试完成的分支，在规划部署时，不能再选择该分支
测试回退的分支，规划部署时，可再次选择该分支
生效中的分支，在规划部署时，不能修改该分支，只能修改失效时间
